## 0. 설계 프레임 — 인풋 → 스코프 → 가공 → 피드백 → 액션

```
[인풋]                  [분석 스코프]              [가공]                 [피드백]            [액션]
사장님 입력(4값·POS)  ─┐   ┌ 가게(자기 데이터)              룰/Z-score          진단·원인        블로그 초안
이미지 OCR            ─┤   ├ 우리 동네(행정동/상권)   →    임베딩/RAG     →    비교·처방    →   광고 세팅
네이버 API(검색·지역) ─┼→ ┤ 시군구(부산내 등)             Prophet/XGB         예측·경보        지원금 신청
공공데이터(상권·날씨) ─┤   └ 전국(업종 평균)               LLM grounded        추적(PDCA)       전문가 연결
웹/리뷰 메타          ─┘
```

**핵심 원칙 두 가지**
1. **데이터마다 작동하는 "스코프"가 다르다.** 출처·스코프를 분리 표기하는 게 정직성이자 차별화.
2. **최종 산출은 "통합 카드 1장".** 우선순위로 선별된 진단 카드, 상위 2개 펴고 나머지 더보기.

**솔루션 택소노미 — 3축**
- **A. 매출↑ / B. 비용↓ / C. 위험↓**
- 진단이 축을 켠다: 매출은 되는데 안 남음→B / 손님 줄어듦→A+C / 위험 스코어 高→C최우선

## 5. 리스크 & 합법성 체크리스트 (v2 §5 유지)

| 항목 | 판단 | 대응 |
|---|---|---|

| **추정매출 데이터** | ❌ 서울 분기 ZIP만 | F1 L3·F4는 서울 한정, 그 외 밀도까지만 |

| 네이버 데이터랩 = 상대값 | 절대 매출 아님 | "추세·신호"로만 표현 |
| 소진공 상가정보 ≠ 비용·매출 | 점포 목록만 | 비용은 업종평균 / 추정매출은 서울 파일 |
| 날씨 상관 없을 수 있음 | 업종별 상이 | 데이터로 측정 후 ON/OFF |

| API 쿼터 초과 | 데모 중 차단 위험 | 사전 캐시 1~2건 |
| 정책 공고 최신성 | 마감일 오류 위험 | "신청 전 원문 확인" 면책 |
